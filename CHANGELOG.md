## V 2.0.0 — 狼人殺模式
> 2026-02-21

加入全新遊戲模式「狼人殺」，架構大幅擴充。

**8 個職業：**
- 🐺 狼人 — 每晚共同選擇獵殺目標
- 👑 狼王 — 被票出局時可帶走一人
- 🔮 預言家 — 每晚查驗一名玩家陣營
- 🧙 女巫 — 解藥（救人）+ 毒藥（殺人）各一瓶
- 🏹 獵人 — 死亡時帶走鎖定目標
- ⚔️ 騎士 — 白天向任意人發起決鬥
- 💣 炸彈客 — 第三方，被全票出局時引爆所有投票者
- 👨‍🌾 村民 — 協助找出並放逐狼人

**遊戲流程：**
- 夜晚：各職業同時進行行動（不互相干擾）
- 白天宣告：公布昨夜死亡玩家
- 討論：自由發言討論
- 投票：選出要放逐的玩家
- 特殊技能：狼王/獵人死後亮牌行動

## V 2.6.0 — 新職業：邱比特 💘
> 2026-02-24

加入第 9 個職業「邱比特」，引入情侶機制與第三方陣營。

**邱比特：**
- 僅在第一個夜晚行動，指定兩名玩家（可包含自己）成為情侶
- 第二夜起自動跳過（不佔用夜晚行動）
- 夜晚有專屬選擇面板（情侶 A / B 卡槽 + 玩家列表）

**情侶系統：**
- 兩位情侶收到私密彈窗通知，告知對方身份
- 一人死亡 → 另一人立刻殉情（不可發動技能）
- 殉情死亡原因顯示「殉情（情侶離去）」

**三種陣營情境：**
| 情況 | 勝利條件 |
|------|----------|
| 兩村民情侶 | 正常村民勝利條件 |
| 兩狼人情侶 | 需屠城（場上≤1名非情侶），判定情侶獲勝 |
| 一狼一村情侶 | 第三方陣營，所有非第三方玩家出局才勝 |

**情侶優先機制：**
- 情侶存活時阻斷所有普通勝利條件（情侶判定優先於所有陣營）
- 情侶雙亡後恢復正常判定
- 結局畫面顯示情侶徽章與第三方勝利橫幅

## V 3.0.0 — 新遊戲：規則混亂 🎲
> 2026-03-25

加入全新遊戲模式「規則混亂」。

**每回合流程：**
1. 所有玩家同時撰寫初始句子 + 制定一條規則
2. 系統隨機抽取一條規則作為本回合唯一規則
3. 系統隨機交換句子（保證不拿到自己的句子）
4. 玩家修改拿到的句子使其符合規則
5. 所有人看到所有原句 → 修改句配對
6. 全體玩家對每位玩家的修改評分（違規／普通／超棒）
7. 計算分數，顯示本回合排行
8. 進入下一回合

**積分制度：** ❌ 違規 −2　👍 普通 +1　⭐ 超棒 +4

**結局統計：** 最終排行榜、超棒次數最多玩家、違規次數最多玩家

**技術：** 獨立 ChaosEngine、7 個相位、Derangement 演算法確保沒人修改自己的句子、主持人控制投票時機、完整房間設定面板

- 改為在 `_listen()` 中呼叫一次 `_watchRoomEmpty()`
- 每個玩家在 `/presence/<pid>` 寫入節點 + `onDisconnect().remove()`
- `_watchRoomEmpty` 監聽 presence 根節點：所有節點消失時自動刪除整個房間

**7. 複製按鈕平滑動畫**

## V 3.1.7 — 獵人亮牌黑屏修復 + 自動鎖定時機修正
> 2026-03-26

**1. 獵人選「亮牌」後全員黑屏修復**

## V 3.1.8 — 故事接龍自動切輪 Bug、狼王修復、獎項揭露
> 2026-03-26

**1. 故事接龍：自動鎖定導致下一輪只跑一秒修復**

根本原因：自動鎖定在 `timeLeft === 1` 時觸發 `checkAllLocked` → 排定 `advance()` 在 +900ms。但計時器繼續跑到 `timeLeft === 0`，此時又排定了 `advance()` 在 +3000ms。第一個 900ms advance 正常切換到新回合並啟動新計時器，但 3000ms 的舊 advance 也跑了一次，把新回合立刻推進。

- `GameEngine._startTimer` 將計時器的 advance 存入 `storyRelay._timerAdvTimer`
- `advance()` 開頭先 `clearTimeout` 兩個 pending timers，再加 phase guard（`if (game.phase !== PHASE.WRITING) return`），確保只執行一次

**2. 狼王：女巫毒死不觸發技能**

狼王被女巫毒殺時，`isWolfkingDeadWithPendingShot` 舊條件只看角色和 `wolfkingSecretReady`，沒有排除女巫毒殺情況。

- 修法二：`_wolfkingSecret` 引擎方法也加同樣守衛

**3. 狼王：秘密帶走面板新增「放棄」按鈕**

死亡覆蓋層的狼王選人面板，在尚未選擇目標時，加入「🤝 放棄，不帶走任何人」按鈕，與獵人和特殊相位面板行為一致。

## V 3.4.1 — 房主斷線踢人 Bug 修復、誤踢 Bug 修復、規則揭曉縮短
> 2026-04-05

**1. 房主斷線導致所有人被踢 — 根本修復**

根本原因：`createRoom()` 對 Firebase 登記了 `rooms/<code>.onDisconnect().remove()`。當 `addPlayer()` 由非房主客戶端呼叫 `onDisconnect().cancel()` 時，此 cancel 只作用在「呼叫者自己的 WebSocket 連線」上，無法取消房主連線上已登記的 onDisconnect。因此，當房主瀏覽器關閉時，Firebase server 仍會執行房主連線上的 `rooms/<code>.remove()`，刪除整個房間（包含所有玩家節點），導致每位玩家的踢人偵測觸發。

修法：
- 完全移除 `rooms/<code>.onDisconnect().remove()`
- 房間清理改由 `_watchRoomEmpty`（監聽 `/players` child_removed，所有玩家都不在才刪房間）負責

**2. 玩家無故顯示「你被房主踢出房間」— 根本修復**

舊實作的踢人偵測：監聽 `rooms/<code>/players/<myId>` 的 `value` 事件，節點消失即視為被踢。此節點在以下情況也會消失，導致誤判：
- Firebase 連線中斷（`onDisconnect` 觸發）→ 重連後 `watchReconnect` 恢復，但這段短暫消失足以觸發
- 房主斷線後整個房間被刪（已由 Fix 1 解決）

新實作：改用專屬的 `/rooms/<code>/kicked/<pid>` 節點：
- 此節點**只有在主持人明確執行踢人操作時才被寫入**
- 斷線、重連、房間結構變動皆不會觸發
- `transport.watchKicked()` 監聽此路徑，`transport.kickPlayer()` 寫入此路徑

**3. 規則揭曉停頓從 5 秒縮短為 3 秒**

`_resolveWriteRule()` 中的倒計時從 `5` 改為 `3`，節奏更緊湊，減少等待感。

## V 3.5.2 — 私訊三項修復 + 規則混亂修改預填原句
> 2026-04-06

**1. 私訊重進後對話內容消失 — 根本修復**

根本原因：`_openDMWith` 每次都無條件停止舊 watcher 並清空訊息，導致按 ← 回到 picker 再點同一個人時，Firebase 重新訂閱並 replay 歷史，但這個過程在視窗關閉期間沒有渲染。

修法：  
- `_openDMWith` 新增 **相同目標短路**：若 `_dmWatcher.targetId === targetId`，直接顯示 convo 並滾到底部，不重建 watcher、不清空訊息
- 只有切換到不同對象時才停止舊 watcher 並清空
- `_closeDMConvo`（← 換人）**不再清空訊息容器**，只切換 UI 視圖，watcher 繼續在背景執行

**2. 私訊紅點提示無法觸發 — 根本修復**

根本原因：`_appendDMMsg` 判斷「是否屬於當前對話」靠 `this._dmTarget`，但 `_closeDMConvo` 把它清為 `null`，所以關掉對話框後所有訊息都被判定為「不屬於任何對話」而跳過渲染與提示。

修法：  
- 採用 `_dmUnreadMap: { pid: count }` 結構，每位聯絡人各自獨立計算未讀數
- `_appendDMMsg` 改用 `_dmWatcher.targetId` 判斷訊息歸屬，不依賴 `_dmTarget`
- picker 列表顯示每位玩家旁的紅色未讀數字徽章（`dm-unread-badge`）
- 進入對話時清空該人的未讀計數並刷新 picker 徽章
- `_updateDMDots` 統一管理：DM tab 上的圓點、面板開啟按鈕上的圓點

**3. 規則混亂修改輸入框預填原句**

進入「修改句子」階段時，輸入框自動填入玩家收到的原始句子，讓玩家直接在原句基礎上修改，而非面對空白框。仍使用 `data-round` key 確保每回合只預填一次，不會干擾已提交或正在編輯的內容。

根本原因：`_show('ww-night-hiddenwolf', amHiddenWolf && !amDone)` 在隱狼 pre-confirm（`nightConfirmed[hwPid] = true`）後 `amDone` 變成 true，面板隱藏，玩家被推入 waiting 場景。
修法：
- 改為 `_show('ww-night-hiddenwolf', amHiddenWolf)`，永遠顯示隱狼專屬面板（類似守墓人）
- `ww-night-waiting` 條件加入 `!(amHiddenWolf && !g.hiddenwolfAwakened)` 排除，未覺醒時不顯示等待畫面

**3. 女巫看不到隱狼刀人目標**
根本原因：女巫 UI 完全依賴 `g.wolfTarget`，但隱狼覺醒後的刀人存放在 `g.hiddenwolfShot`，女巫畫面永遠顯示「等待狼人確認目標中」。
修法：新增 `effectiveKillTarget = g.wolfTarget || (g.hiddenwolfAwakened && g.hiddenwolfShot ? g.hiddenwolfShot : null)`，女巫看到的是受害者本人（不知道是誰刀的），保持身份隱匿。同時 `witchSave` 現在也能阻擋隱狼刀人（`_resolveNight` 加入 `&& !g.witchSave` 條件）。

**4. 隱狼在設定面板位置移至狼人與狼王之間**
調整 `index.html` 角色計數器順序：狼人 → 隱狼 → 狼王 → 其他職業。

**5. 私訊：直接關閉聊天視窗（未按 ← 換人），重開後訊息不顯示**
根本原因：關閉按鈕（✕）只設 `_chatOpen = false`，watcher 仍在背景運作。新訊息抵達時 `_appendDMMsg` 判斷 `convoVisible = false`（面板關閉），訊息不寫入 DOM，只計未讀。重開時因 watcher 仍存在觸發「相同目標短路」，直接 scrollTop 而非重載，舊 DOM 缺少訊息。
修法：
- `closeBtn` 點擊時若在 DM 對話中，停止 watcher（保留 `_dmTarget` 記錄位置）
- `openBtn` 重開時若 `_dmTarget !== null`，呼叫 `_openDMWith` 做完整 Firebase reload（`limitToLast(80)` 帶回所有訊息含關閉期間新訊息）

## V 3.6.2 — 隱狼三項修復 + 私訊紅點通知系統重建
> 2026-04-12

**Bug 1：隱狼未覺醒時面板是黑畫面**

根本原因：渲染條件為 `if (amHiddenWolf && !amDone)`，但 `_startNight` 中隱狼被 pre-confirm（`nightConfirmed[hwPid] = true`），使 `amDone = true`，導致整個內容區不渲染 → 黑畫面。

修法：條件改為 `if (amHiddenWolf)`，並在內部以三個分支處理所有狀態：
- **未覺醒（amDone=true but not awakened）**：顯示狼隊友狀態列表 + 月亮互動玩具
- **覺醒且未確認**：顯示震撼覺醒橫幅 + 選人刀格 + 確認/放棄按鈕
- **覺醒且已確認**：顯示「✓ 已確認行動，等待天亮」

**Bug 2：隱狼無法使用確認鍵讓女巫看到狼刀**

根本原因：`effectiveKillTarget` 使用 `g.hiddenwolfShot`（選擇目標時就設定），導致女巫在 hw 按下晶片的瞬間就看到受害者，且確認按鈕因 `disabled` 邏輯問題無法正常使用。

修法：`effectiveKillTarget` 改為 `g.hiddenwolfDone && g.hiddenwolfShot`，只有 hw 按下「確認刀人」後，女巫才能看到被刀目標（身份仍隱匿）。

**Bug 3：私訊紅點通知系統（全面重建）**

問題：收到私訊時不顯示紅點；直接關閉聊天視窗後重開無法顯示通知。

根本原因（雙重）：
1. 現有程式碼完全沒有「對話外通知」機制——私訊通知完全依賴 `watchDM`，若沒有開過某人的對話就沒有 watcher，訊息到達時無任何機制觸發紅點。
2. 直接關閉視窗（不按 ← 換人）時 `_dmTarget` 保留，`_activeTab` 仍為 `'dm'`，reopening 時重進同一對話，新訊息若在關閉期間到達因 `convoVisible=false` 而不寫 DOM，同時 `_dmUnreadMap` 也不計數。

修法（三層）：
- **新增 `dm_inbox` 通知路徑**（Firebase `/dm_inbox/<toId>/<fromId>`）：傳送時寫入（`writeDMInbox`），進入對話時清除（`clearDMInbox`），進房間後以 `startAt(joinTs)` 啟動監聽（`watchDMInbox`），避免歷史殘留通知。
- **`closeBtn` 關閉時立即導正**：切換 `_activeTab='public'`、停止 DM watcher、重置 `_dmTarget=null`、UI 切回公開聊天、呼叫 `_updateDMDots()` 更新紅點。重開面板時自然落在公開聊天，DM 紅點正確顯示。
- **`openBtn` 修正**：移除無條件清除 `dm-tab-dot` 的行為，改由 `_updateDMDots()` 根據實際未讀狀態決定。
